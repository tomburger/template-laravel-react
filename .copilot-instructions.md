# Copilot Instructions - Laravel + React Template

## Project Overview
This is a headless architecture template combining:
- **Backend**: Laravel (REST API only, no views)
- **Frontend**: React (standalone UI consuming the API)

## Technology Stack

### Backend (Laravel)
- **Framework**: Laravel (latest stable)
- **API Style**: RESTful JSON APIs + Scribe to generate OpenAPI documentation
- **Database**: no dependencies, rely on Laravel's Eloquent ORM for database interactions
- **Authentication**: Laravel native for APIs
- **User Management**: Built-in user model with authentication scaffolding

### Frontend (React)
- **Framework**: React 18+
- **Language**: TypeScript
- **Build Tool**: Vite
- **HTTP Client**: generated from OpenAPI spec using Orval
- **Styling**: Bootstrap (not Tailwind!)

## Coding Standards

### PHP / Laravel
- **PHP Version**: 8.1+
- **Style**: PSR-12 coding standard
- **Naming**: 
  - Controllers: `UserController`, `ProductController` (singular)
  - Models: `User`, `Product` (singular)
  - Routes: kebab-case (`/api/users`, `/api/user-profiles`)
  - Methods: camelCase
- **API Responses**: Consistent JSON structure with status codes
- **Error Handling**: Return meaningful error messages with appropriate HTTP status codes

### TypeScript / React
- **Version**: ES6+ (modern JavaScript)
- **Style**: Consistent with Prettier/ESLint configuration
- **Naming**:
  - Components: PascalCase (`UserList`, `ProductCard`)
  - Functions/variables: camelCase
  - Constants: UPPER_SNAKE_CASE
  - Files: Match component names (e.g., `UserList.tsx`)
- **Functional Components**: Prefer hooks over class components

## API Design

### Conventions
- All API routes prefixed with `/api/` (e.g., `/api/v1/`)
- RESTful endpoints (GET, POST, PUT/PATCH, DELETE)
- JSON request/response format
- Consistent error response structure
- CORS configured for frontend domain

### Example Response Structure
```json
{
  "status": "success",
  "data": { /* payload */ },
  "message": "Optional message"
}
```

### Error Response Structure
```json
{
  "status": "error",
  "message": "Error description",
  "errors": { /* validation errors */ }
}
```

## Authentication & Authorization
- API endpoints use stateless authentication (JWT or token-based)
- User credentials are managed in Laravel
- Frontend stores tokens and includes in API request headers
- CORS headers configured appropriately

## Development Workflow

### Backend Development
- Create models and migrations together
- Write API tests for new endpoints
- Use Laravel Artisan commands for scaffolding
- Document API endpoints

### Frontend Development
- Create reusable React components
- Use hooks for state management
- Handle loading and error states
- Call API endpoints via configured HTTP client
- Prefer Vite live watching for validation during normal development
- Do not run `npm run build` unless the user explicitly asks for a production build check

## Key Guidelines for Code Generation

1. **Separation of Concerns**: Keep backend logic (validation, business rules) separate from frontend (UI, UX)
2. **Reusability**: Create reusable components and API helper functions
3. **Error Handling**: Always handle API errors gracefully in the UI
4. **Testing**: Include unit tests for components and API endpoints
6. **Security**: 
   - Validate input on both frontend and backend
   - Use HTTPS in production
   - Secure CORS configuration
   - Implement rate limiting on API
   - Never expose sensitive keys in frontend code

## File Naming Conventions
- Laravel: `UserController.php`, `CreateUsersTable.php`
- React: `UserProfile.tsx`, `useUserData.ts` (custom hooks)
- Config files: `.env`, `*.config.js`, `*.json`

## Recommended Project Setup
- npm for frontend dependencies
- Composer for backend dependencies
- Git for version control with meaningful commits
- `.env` files for configuration (not committed to repo)
- Vite for local development with hot reloading
